## Summary

0.3.0 splits the ai-setup into a **shareable layer** (`ai/developer.agent.md`, `ai/developer.instructions.md`,
`ai/spec.md`, plugin overlays) and a **private/local layer** (`ai/memory/`). `developer.instructions.md`
moves out of `ai/memory/` up to `ai/`, beside `developer.agent.md`, and becomes shareable: it must not carry
hostnames, IPs, internal/corporate URLs, deploy targets, or secrets. This migration relocates the file and
then drives a **lossless** content scrub - any such detail is moved **into** `ai/memory/` (resources /
credentials / a project `info.md`), never deleted. (`developer.instructions.md` is not revision-tracked, so
the move does not touch the revisions ledger or baseline; the path edits in `developer.agent.md` and
`AGENTS.md` are reconciled by `update-project`'s revisions step, which runs before this migration.)

## Pre-flight checks

- [ ] `projects/<slug>/ai/manifest.json` exists and is valid JSON.
- [ ] The framework is at 0.3.0 (`uv run -m solaris.tools.version current`).
- [ ] `projects/<slug>/ai/memory/developer.instructions.md` exists (else the move was already done - skip step 1).

## Migrate

1. **Relocate the file** (only if not already moved): move
   `projects/<slug>/ai/memory/developer.instructions.md` -> `projects/<slug>/ai/developer.instructions.md`
   (`git mv` when the project's `src/` git or an ai/ git tracks it; otherwise a plain move). If
   `ai/developer.instructions.md` already exists, skip - already migrated.
2. **Fix in-file relative links** now that the file lives one level up: `../developer.agent.md` ->
   `developer.agent.md`; any sibling pointer such as `info.md` -> `memory/info.md`.
3. **Lossless scrub (editorial - agent + user).** Read `ai/developer.instructions.md` and, for every
   hostname, IP, tailnet/VPN address, internal/corporate URL, deploy target, remote path, cluster/node name,
   or secret, **relocate it** into the right private file and replace it in place with a generic placeholder
   plus a pointer:
   - hosts / IPs / URLs / deploy target / remote path / ports-tied-to-a-host -> `ai/memory/resources.md`
     (correct it if stale). Commands become generic patterns, e.g. `--host <host> --port <port>` /
     `BASE=http://<host>:<port> ...`, with "values in `ai/memory/resources.md`".
   - secrets / keys / env values -> `ai/memory/credentials.md` (reference, never paste).
   - infra facts / dataset locations / sample node lists -> `ai/memory/resources.md` or a project
     `ai/memory/info.md`.
   Nothing is deleted - every removed detail must end up in a memory file. Generic build/run/test commands,
   toolchain quirks, and conventions stay in `ai/developer.instructions.md`.
4. **Stamp the version:** `uv run -m solaris.tools.version set --dir projects/<slug> 0.3.0`.

## Validate

- [ ] `projects/<slug>/ai/developer.instructions.md` exists; `ai/memory/developer.instructions.md` does not.
- [ ] No hostname, IP, internal URL, or secret remains in `ai/developer.instructions.md` (spot-check); each
      relocated detail is present in `ai/memory/` (resources / credentials / info).
- [ ] `ai/manifest.json` `framework_version` is `0.3.0` (`uv run -m solaris.tools.version current --dir projects/<slug>`).
- [ ] `uv run -m solaris.tools.revs classify --dir projects/<slug>` reports all `in-sync` (the
      `developer.agent.md` / `AGENTS.md` path edits were applied by `update-project`'s revisions step).

## Revert

1. `uv run -m solaris.tools.version set --dir projects/<slug> 0.2.0`.
2. Move `ai/developer.instructions.md` back to `ai/memory/developer.instructions.md` and restore its in-file
   relative links (`developer.agent.md` -> `../developer.agent.md`, `memory/info.md` -> `info.md`).
   (The scrub is not auto-reversed: details relocated into `ai/memory/` simply remain there - a harmless
   duplication under 0.2.0. Re-inline them by hand only if you specifically want the old single-file layout.)
